1. Create a remote machine with hardware ciretualization enabled.
Check it using:
ls -l /dev/kvm.
groups | grep kvm # Your user (or the service user) should be in kvm group
Output: Ensure it returns a character device (crw-rw----) owned by root and the kvm group. If it fails, Firecracker will instantly crash on boot.

2. Install firecracker and remember its path.

3. Install Kata Contianer with QEMU and remember its path.

4. Now, we need to join pipes btw Kata Continaer and Firecracker. We do it by giving firecracker binary we remembered to Kata and remove QEMU.
i) cerate folder for new config file. By default kata search config file in /etc/kata-containers/configuration.toml and fall is some /.../share/.../configuration-qeumu.toml:
sudo mkdir /etc/kata-containers
ii) Bring configuration-fc.toml to active folder:
sudo cp /opt/kata/share/defaults/kata-containers/configuration-fc.toml /etc/kata-containers/configuration-fc.toml
iii) Modify or Verify path of firecracker binary in configuration-fc.toml using:
nano /etc/kata-containers/configuration-fc.toml
and ensure the [hypervisor.firecracker] section has the exact path  = which firecracker.
Vi) We need to use configuration-fc.toml so we create symlink (which redirects file from hardcoded configuration.toml to configuration-fc.toml):
sudo ln -s /etc/kata-containers/configuration-fc.toml /etc/kata-containers/configuration.toml
V) Map hardcoded file path for containerd-shim-kata-v2 to actaul path:
sudo ln -sf /opt/kata/bin/* /usr/local/bin/
Check:
sudo kata-runtime check (This should output a success message saying the host is capable of running Kata).
sudo kata-runtime kata-check

5. Start k3s cluster with zero config: sh -

6. Now plumbing btw contianerd and Kata container:
i) create a template file:
sudo cp /var/lib/rancher/k3s/agent/etc/containerd/config.toml /var/lib/rancher/k3s/agent/etc/containerd/config-v3.toml.tmpl
ii) Modify file to detect runtimeclass = 'kata-fc'. By adding plugin and pointing Kata inside it:
sudo nano /var/lib/rancher/k3s/agent/etc/containerd/config-v3.toml.tmpl
[plugins."io.containerd.cri.v1.runtime".containerd.runtimes.kata-fc]
  runtime_type = "io.containerd.kata.v2"
  privileged_without_host_devices = true
  pod_annotations = ["io.katacontainers.*"]
  
[plugins."io.containerd.cri.v1.runtime".containerd.runtimes.kata-fc.options]
  ConfigPath = "/etc/kata-containers/configuration.toml"

iii)  sudo systemctl restart k3s
iv) Check: sudo crictl info | grep kata-fc (If this returns nothing, the plumbing failed. If it returns the kata-fc block, containerd is locked and loaded).

7. Now, informing the k3s cluster Brain/API server about existence of 'kata-fc' container, so that it do not reject the request:
i) API call request to create:
cat <<EOF | kubectl apply -f -
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: kata-fc
handler: kata-fc
EOF
ii) Check: kubectl get runtimeclass
kata-fc should be listed now.

8. Now I can start the k3s cluster and call API server using curl to create a microVM using runtimeclass: kata.
kubectl run kata-pod --image=nginx --overrides='{"spec":{"runtimeClassName":"kata-fc"}}' --restart=Never
Check: kubectl get pods -w
If it gets stuck in ContainerCreating forever, DO NOT look at Kubernetes logs. A hang here means your Step 4 (Kata -> Firecracker) or Step 5 (containerd -> Kata) plumbing has a typo or a broken symlink.
