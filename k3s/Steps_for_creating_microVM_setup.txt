1. Create a remote machine with hardware ciretualization enabled.
Check it using:
ls -l /dev/kvm.
groups | grep kvm # Your user (or the service user) should be in kvm group
Output:
i) Ensure it returns a character device (crw-rw----) owned by root and the kvm group.
ii) You can also run kata-runtime env to verify the hypervisor path is pointing to QEMU

2. Install Kata Contianer with QEMU and remember its path.
Check:
sudo kata-runtime check (This should output a success message saying the host is capable of running Kata).
sudo kata-runtime kata-check
sudo kata-runtime env

3. Start k3s cluster with zero config:
curl -sfL https://get.k3s.io | sh -

4. Now plumbing btw k3s embedded contianerd and Kata container:
i) create a template file:
sudo cp /var/lib/rancher/k3s/agent/etc/containerd/config.toml /var/lib/rancher/k3s/agent/etc/containerd/config.toml.tmpl
ii) Modify file to detect runtimeclass = 'kata-qemu'. By adding plugin and pointing Kata inside it:
sudo nano /var/lib/rancher/k3s/agent/etc/containerd/config.toml.tmpl
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.kata-qemu]
  runtime_type = "io.containerd.kata.v2"
  privileged_without_host_devices = true
  pod_annotations = ["io.katacontainers.*"]
  
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.kata-qemu.options]
  ConfigPath = "/opt/kata/share/defaults/kata-containers/configuration-qemu.toml"

Configpath should be correct, check by using: sudo kata-runtime env

iii)  sudo systemctl restart k3s
iv) Check: sudo crictl info | grep kata-qemu (If this returns nothing, the plumbing failed. If it returns the kata-qemu block, containerd is locked and loaded).

5. Now, informing the k3s cluster Brain/API server about existence of 'kata-qemu' container, so that it do not reject the request:
i) API call request to create:
cat <<EOF | kubectl apply -f -
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: kata-qemu
handler: kata-qemu
EOF
ii) Check: kubectl get runtimeclass
kata-qemu should be listed now.

6. Now I can start the k3s cluster and call API server using curl to create a microVM using runtimeclass: kata.
kubectl run kata-pod --image=nginx --overrides='{"spec":{"runtimeClassName":"kata-qemu"}}' --restart=Never
Check: kubectl get pods -w
If it gets stuck in ContainerCreating forever, DO NOT look at Kubernetes logs. A hang here means your Step 4 (Kata -> Firecracker) or Step 5 (containerd -> Kata) plumbing has a typo or a broken symlink.

7. Use port forawarding btw to acces microVM nginx using local machine via remote Host Kernel.
